

# Wire Frontend Shell to Live Supabase Backend

Replace all mock data with real Supabase queries while keeping every existing route, layout, component, and design token unchanged.

## Prerequisite (one click from you)

The Lovable project is not yet linked to your Supabase project — I need the URL and publishable (anon) key to be available as env vars before any code can talk to your database. **Click the green Supabase button in the top-right of the editor and select your project.** That auto-injects `VITE_SUPABASE_URL` and `VITE_SUPABASE_PUBLISHABLE_KEY` and creates `src/integrations/supabase/`. Once linked, I'll execute everything below in one pass — no further action from you.

## What gets built

### 1. Supabase client + typed schema
- `src/integrations/supabase/client.ts` — browser client (auto-generated by the integration)
- `src/integrations/supabase/types.ts` — generated TypeScript types from your live schema

### 2. Auth context + protected routes
- `src/lib/auth.tsx` — `AuthProvider` exposing `session`, `user`, `profile`, `currentOrg`, `loading`. Uses `onAuthStateChange` first, then `getSession()` (correct order — prevents stale state).
- `src/routes/__root.tsx` — wrap `<Outlet />` with `AuthProvider` and `QueryClientProvider`.
- `src/router.tsx` — fresh `QueryClient` per request, register context type.
- `src/routes/_authenticated.tsx` — pathless layout with `beforeLoad` that redirects to `/auth/sign-in` if no session and stashes `redirect=` for return-after-login.
- Move existing `app.*.tsx` routes under the `_authenticated` boundary (rename to `_authenticated.app.*.tsx`) so every `/app/*` URL is gated. URLs and components stay identical.

### 3. Auth flows wired to Supabase
- `auth.sign-in.tsx` → `supabase.auth.signInWithPassword`, redirect to stashed URL or `/app/dashboard`.
- `auth.sign-up.tsx` → `supabase.auth.signUp` with `emailRedirectTo: window.location.origin`, then routes to `/onboarding`.
- `auth.forgot-password.tsx` → `resetPasswordForEmail` with `redirectTo: origin + '/auth/reset-password'`.
- `auth.reset-password.tsx` → detects recovery hash, calls `updateUser({ password })`.
- Topbar logout button → `supabase.auth.signOut()` then redirect to `/auth/sign-in`.

### 4. Onboarding → real persistence
- `onboarding.tsx` saves answers per step into `onboarding_responses`.
- On finish: insert into `organizations`, insert membership into `organization_members` as `owner`, update `profiles.full_name`, insert default `subscriptions` row with `plan = 'starter'`, then route by stage (Operate/Scale → `/app/nova`, else `/app/launchpad`).

### 5. Live data replacing `src/lib/mock.ts`
Build `src/lib/queries.ts` with `queryOptions` for each entity. Each `/app/*` route gets a `loader` that calls `queryClient.ensureQueryData(...)` and components read via `useSuspenseQuery`. Replacements:

| Screen | Was (mock) | Now (Supabase) |
|---|---|---|
| Dashboard | `currentUser`, `currentCompany`, `recommendedNext` | `profiles` + active `organizations` + computed next step from `tool_runs` |
| Launchpad index | `launchpadTools`, `recentOutputs` | Static tool catalog (kept in code) + `tool_runs` joined with `generated_assets` |
| Launchpad `$tool` | mock generation | Insert `tool_runs` row (status `running` → `succeeded`), insert outputs into `generated_assets`, "Send to Nova" sets `metadata.sent_to_nova = true` |
| Launchpad history | `recentOutputs` | `tool_runs` ordered by `created_at desc` |
| Nova CRM / Leads / Workflows / Clients / Reports | static arrays | Read from `tool_runs` + `generated_assets` filtered by `category` for the current org. Where your schema doesn't have CRM/leads tables yet, I'll persist these as `generated_assets` rows with a typed `kind` until you add dedicated tables. |
| Billing | `plans`, `usage` | `subscriptions` for current state, `plan_entitlements` for feature/limit display. Buttons change plan in `subscriptions` (no payment processor — visual only, matches existing UX). |
| Settings | static | `profiles` (name/email read-only) + `organizations` (name, niche, stage, etc.) with update mutations |
| Stage tracker | hardcoded | `organizations.stage` |
| Locked overlay | hardcoded plan | Compares `subscriptions.plan` against `plan_entitlements.required_plan` |

All mutations call `queryClient.invalidateQueries(...)` on success so the UI refreshes.

### 6. Multi-tenant scoping
A single `useCurrentOrg()` helper resolves the active org via `organization_members` for `auth.uid()` (first owned org, persisted to `localStorage` for switching later). Every query filters by `organization_id`. RLS policies on your tables already enforce this server-side — the client filters are for correctness and performance.

### 7. Design system + UI
**Zero changes.** No edits to `styles.css`, `components/ui/*`, `AppSidebar`, `AppTopbar`, `PageHeader`, `StageTracker`, `LockedOverlay`, or any route's JSX structure. Only the data source behind props changes.

## Technical details

- **TanStack Query**: `QueryClient` created inside `getRouter` factory (SSR-safe), `defaultPreloadStaleTime: 0` already set. Loaders `ensureQueryData`, components `useSuspenseQuery`.
- **Auth in router context**: `createRootRouteWithContext<{ queryClient, supabase }>()`. Auth state lives in React context (not router context) because it changes over time — `_authenticated.beforeLoad` calls `supabase.auth.getSession()` directly.
- **No server functions needed**: All reads/writes are RLS-gated client calls. No `createServerFn`, no edge functions, no service role key in the project.
- **Type safety**: `Database` type from `types.ts` flows through every query. No `any`.
- **File deletions**: `src/lib/mock.ts` stays as a fallback type reference for the static Launchpad tool catalog (names, descriptions, icons), but all dynamic arrays are removed.
